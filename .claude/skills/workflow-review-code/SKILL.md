# コードレビューワークフロー

現在のブランチの変更差分を4つの専門観点で並列レビューし、
優先度付きのフィードバックと対応方針を提示するワークフロー。

## Phase 1: 差分収集

レビュー対象の差分を収集する。

```bash
# main ブランチとの差分ファイル一覧
git diff --name-only main...HEAD

# 差分の詳細
git diff main...HEAD

# コミット履歴
git log main...HEAD --oneline
```

差分がない場合は「レビュー対象の変更がありません」と報告して終了する。

## Phase 2: 並列レビュー

4つのサブエージェント（Agent ツール）を**同一ターンで並列起動**する。
各エージェントには差分ファイル一覧と変更内容のコンテキストを渡す。

各エージェントへの指示には以下を含める:
- レビュー対象のブランチ名と main からの差分ファイル一覧
- `git diff main...HEAD` で差分を読み、関連する既存コードも Read で確認すること
- フィードバックは Phase 2 末尾の出力フォーマットに従うこと
- レビューのみ実行し、コードの修正は行わないこと

---

### エージェント 1: 設計・品質レビュー

プロジェクトの設計方針との整合性、エラーハンドリング、コードの明確さを検証する。

#### 設計遵守

- **依存方向**: `presentation -> application -> domain <- infrastructure` が守られているか
  - domain 層がフレームワーク（Hono, DB, MinIO）に依存していないか
  - infrastructure が domain のインターフェースを実装しているか（依存性逆転）
- **DI パターン**: 高階関数による DI が使われているか（IoC コンテナ・デコレータは不可）
- **CQRS**: 読み書きの責務が分離されているか
  - Query が EventStore やドメインモデルを経由していないか
  - Command がバリデーションとイベント生成に集中しているか
- **イベントソーシング**: コマンド関数がイベントを返し、applyEvent で状態を適用しているか
- **フロントエンド分離**: UI (`xxx.tsx`) / ロジック (`xxx.hooks.tsx`) が分かれているか
- **Blob パス規約**: `pending/` -> `confirmed/` フローが守られているか
- **ID 規約**: uuid v7 + base62 エンコードが使われているか

#### エラーハンドリング

- Result 型（`ok: true / ok: false`）が一貫して使われているか
- システム境界（API エンドポイント）でバリデーションされているか
- ドメイン層でビジネスルール違反が適切にエラーとして表現されているか

#### コードの明確さ

- 意図が読み取れるか、不要な複雑さがないか
- 命名が適切か（ドメイン用語との一致）
- 不要な抽象化や過剰な設計がないか

参照すべきドキュメント:
- `CLAUDE.md` の制約事項セクション
- `docs/guides/architecture/backend.md`
- `docs/guides/architecture/frontend.md`
- `docs/guides/event-sourcing.md`

### エージェント 2: セキュリティレビュー

OWASP Top 10 とプロジェクト固有のセキュリティ制約を検証する。

- **インジェクション**: SQL インジェクション、コマンドインジェクション、XSS の可能性
  - パラメータ化クエリが使われているか
  - ユーザー入力がサニタイズされているか
- **認証・認可**: APS Client Secret がフロントエンドに露出していないか
- **Blob ストレージ**: アクセス制御が適切か
  - `pending/` prefix のファイルが未確認のまま公開されないか
  - パストラバーサルの可能性がないか
- **入力バリデーション**: システム境界（API エンドポイント）で zod スキーマが適切に定義されているか
- **機密情報**: 環境変数・シークレットがハードコードされていないか
- **依存パッケージ**: 既知の脆弱性がある依存が追加されていないか

---

### エージェント 3: テスト過不足レビュー

テスト戦略 (`docs/guides/testing.md`) との整合性と、テストの過不足を検証する。

- **ドメイン層**: 変更された値オブジェクト・エンティティ・コマンド関数にテストがあるか
  - 正常系・異常系・境界値の3観点がカバーされているか
  - Result 型の ok: true / ok: false 両方がテストされているか
  - イベントソーシングのコマンド -> イベント -> 状態適用の一連がテストされているか
- **インフラ層**: Repository 実装の変更に結合テストがあるか
  - ラウンドトリップ（save -> load）がテストされているか
  - テストヘルパー・フィクスチャが適切に使われているか
- **テスト品質**: テストが実装の詳細ではなく振る舞いをテストしているか
- **テスト配置**: テストファイルが実装と同階層にコロケーションされているか（`__tests__/` は不可）
- **過剰テスト**: 不要なテスト（API エンドポイントの単体テスト等、テスト対象外のもの）が含まれていないか
- **カバレッジ**: ドメイン層は 100% を目標としているが、未カバーのパスがないか

参照すべきドキュメント:
- `docs/guides/testing.md`

---

### エージェント 4: 実装の実効性レビュー

コードが正しく動作するか、効率的かを検証する。設計パターンへの準拠ではなく、実際の振る舞いに着目する。

#### ビジネスロジックの正しさ

- エッジケースの考慮漏れがないか（null、空配列、境界値）
- 状態遷移のルールが正しく実装されているか
- イベントの順序やバージョン管理に不整合がないか
- 非同期処理の競合状態（楽観的同時実行制御の漏れ等）がないか

#### パフォーマンス

- N+1 クエリが発生していないか
- 不要な全件取得がないか（適切な WHERE / LIMIT の使用）
- インデックスが必要なカラムにインデックスが定義されているか
- 不要なデータの読み込み（SELECT * の濫用）がないか
- イベントストアからの rehydrate でイベント数が多い場合の性能考慮があるか

### 各エージェントの出力フォーマット

各エージェントは以下の形式で結果を返すこと:

```
観点名: [設計・品質 / セキュリティ / テスト過不足 / 実装の実効性]
件数: N件

---

[P0] ファイルパス:行番号
内容: 何が問題か
理由: なぜ問題か
---

[P1] ファイルパス:行番号
内容: 何が問題か
理由: なぜ問題か
---

(以下同様)
```

指摘がない場合は「指摘なし」と返す。

優先度の定義:
- **P0（即時対応）**: セキュリティ脆弱性、データ破損リスク、アーキテクチャ違反（依存方向の逆転など）、ロジックバグ
- **P1（対応推奨）**: 設計方針からの逸脱、テストの欠落、パフォーマンス問題、エラーハンドリング不足
- **P2（検討）**: 改善提案、ベストプラクティスの推奨、軽微な指摘

## Phase 3: 結果集約・提示

4つのエージェントの結果を統合し、以下の形式でユーザーに提示する。

```markdown
## コードレビュー結果

### サマリー
| 観点 | P0 | P1 | P2 | 合計 |
|------|----|----|----|----|
| 設計・品質 | X | X | X | X |
| セキュリティ | X | X | X | X |
| テスト過不足 | X | X | X | X |
| 実装の実効性 | X | X | X | X |
| **合計** | **X** | **X** | **X** | **X** |

### P0（即時対応）
（P0 の指摘を全て列挙。なければ「なし」）

1. **[観点] ファイルパス:行番号**
   - 内容: ...
   - 理由: ...

### P1（対応推奨）
（同様に列挙）

### P2（検討）
（同様に列挙）
```

指摘が0件の場合は「指摘事項はありません。コードは設計方針に沿っており、品質に問題はありません。」と報告する。

## Phase 4: 対応方針の提案と承認

指摘が1件以上ある場合、対応方針を提案してユーザーの承認を求める。

```markdown
## 対応方針

### 対応する指摘
| # | 優先度 | 指摘内容（要約） | 対応方針 |
|---|--------|-----------------|---------|
| 1 | P0 | ... | 具体的な修正内容 |
| 2 | P1 | ... | 具体的な修正内容 |

### 対応しない指摘（あれば）
| # | 優先度 | 指摘内容（要約） | スキップ理由 |
|---|--------|-----------------|-------------|
| 3 | P2 | ... | 現時点では過剰 |
```

P0 は全て対応に含める。P1 は原則全て含める。P2 はコスト対効果を考慮して判断する。

**ユーザーの承認なしに修正を開始しないこと。**

承認後は対応方針に沿って修正を実施する。
