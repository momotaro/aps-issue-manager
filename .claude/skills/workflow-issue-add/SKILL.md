# Issue 作成ワークフロー

ユーザーの自然言語リクエストから GitHub Issue を作成するための全手順。
デザイン承認と Issue 内容承認の2段階で確認を取り、承認なしに Issue を作成しないこと。

## Phase 1: 調査

ユーザーのリクエストを元にコードベースと設計ドキュメントを調査する。

### 調査対象
- `docs/guides/architecture.md` — 既存アーキテクチャとの整合性
- `CLAUDE.md` — プロジェクトの制約事項
- 関連する既存コード — Explore エージェントで探索し、影響範囲を把握
- 既存 Issue — `gh issue list --state all` で重複がないか確認

### 調査で明らかにすること
- バックエンドタスクの有無と内容（どのレイヤーに変更が必要か）
- フロントエンドタスクの有無と内容（どのコンポーネントが関わるか）
- DB マイグレーションの要否
- 完了の定義として検証可能な条件

## Phase 2: 実装計画

調査結果を元に Issue の構成案を立て、ユーザーに方向性を共有する。

### 明確にする内容
- **フロントエンド作業の有無** → デザイン作成（Phase 3）の要否を判定
- **バックエンドのレイヤー別タスク**:
  - domain（エンティティ、値オブジェクト、リポジトリインターフェース）
  - application（ユースケース）
  - infrastructure（リポジトリ実装、外部サービス連携）
  - presentation（ルート定義、クエリ/ミューテーション）
- **フロントエンドのタスク**:
  - サーバーコンポーネント（RSC）
  - クライアントコンポーネント（CC）
  - カスタムフック
- **完了の定義** — 具体的で検証可能な条件をリストアップ
- **DB マイグレーションの要否**

この計画をユーザーに共有し、方向性の合意を得てから次に進む。
合意なしに Phase 3 に進まないこと。

## Phase 3: デザイン作成

フロントエンド作業がある場合のみ実行する。ない場合は Phase 5 にスキップ。

### Pencil MCP によるデザイン作成

1. `get_guidelines` でプロジェクトのスタイルガイドを確認
2. `open_document('new')` で新規 .pen ファイルを作成
   - または既存の関連 .pen ファイルがあれば `open_document(filePath)` で開く
3. `batch_design` でデザインを作成
   - Phase 2 の計画に沿ったコンポーネント構成にする
   - スタイルガイドに準拠する
4. `get_screenshot` でプレビューを取得

### 保存先・運用ルール
CLAUDE.md「デザイン（.pen）」の制約に従う。

- `page.pen` を `page.tsx` と同じディレクトリに配置（ページ単位）
- ノード名は実装コンポーネント名に合わせる

例: `frontend/src/app/issues/page.pen`

## Phase 4: デザイン承認

デザインのスクリーンショットをユーザーに提示し、承認を得る。

- 修正が必要な場合は Phase 3 に戻って修正
- ユーザーが承認したら Phase 5 に進む
- フロントエンド作業がない場合はスキップ

## Phase 5: Issue 内容生成

`docs/templates/issue_feature.md` のフォーマットに沿って Issue body を生成する。
テンプレートの各セクションを埋める:

```markdown
# 📊 Issue: [機能名]

### 🎯 目的
- 誰が / 何を / なぜ（ユーザーのリクエストから）

### 背景
- 現状の課題（調査結果から）

### ⚙️ 完了の定義
- [ ] 検証可能な条件（Phase 2 の計画から）

---

## 🛠️ バックエンド タスク（該当する場合）
### 1. アプリケーション層
### 2. ドメイン層
### 3. インフラストラクチャ層
### 4. プレゼンテーション層

---

## 🖥️ フロントエンド タスク（該当する場合）
### 1. サーバーコンポーネント
### 2. クライアントコンポーネント
### 3. カスタムフック

---

## 🎨 デザイン（.pen ファイルがある場合）

| ファイル | 説明 |
| :--- | :--- |
| `frontend/src/app/issues/_components/IssueFilter.pen` | 対象コンポーネント |

---

## 📝 補足
```

### テンプレート遵守が重要な理由

この Issue は `workflow:feature` スキルの `fetch-issue.mjs` でパースされる。
以下のセクションヘッダーが正確に一致しないとパースが失敗する:
- `## 🛠️ バックエンド` — バックエンドタスク検出
- `## 🖥️ フロントエンド` — フロントエンドタスク検出
- `### ⚙️ 完了の定義` — 完了の定義抽出
- `## 🎨 デザイン` — .pen ファイルパス抽出

タスクはテーブル形式（`| No. | 対応内容 |`）で記述すること。

## Phase 6: Issue 承認

生成した Issue 内容の全文をユーザーに提示し、承認を得る。

- 修正が必要な場合は Phase 5 に戻って修正
- ユーザーが承認したら Phase 7 に進む

## Phase 7: GitHub に作成

スクリプトを実行して Issue を作成し、GitHub Projects に追加する。

### Issue タイトル
- 形式: `[機能名]`（短く簡潔に）

### スクリプト実行
```bash
echo '<Issue body>' | node .claude/skills/workflow-issue-add/scripts/create-issue.mjs \
  --title "<Issue タイトル>" \
  --label feature
```

スクリプトが以下を実行:
1. `gh issue create` で Issue 作成
2. `gh project item-add 4` で GitHub Projects #4 に追加
3. 結果 JSON を出力

### 完了報告
作成された Issue の URL をユーザーに報告する。
